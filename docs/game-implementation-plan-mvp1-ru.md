# План реализации MVP-1: базовый цикл и колода персонажей

## 1. Цель этапа

Реализовать минимально рабочий игровой движок:
- матч;
- раунды;
- ходы игроков;
- базовые карты персонажей;
- завершение раунда и начисление ПО.

Что исключено на этом этапе:
- указы;
- события;
- карта `Знамение`;
- токены и модификаторы значений от указов/событий.

## 2. Скоуп правил MVP-1

### Включить

- Подготовка раунда:
- перемешать колоду персонажей без `Тени` и без `Знамения`;
- отложить одну карту;
- раздать по 1 карте игрокам;
- определить первого игрока/лидера по текущему правилу.

- Ход игрока:
- добрать карту;
- разыграть одну карту;
- применить базовый эффект карты персонажа;
- передать ход следующему живому игроку.

- Завершение раунда:
- если остался один живой игрок;
- если исчерпана колода персонажей после завершения текущего хода.

- Подсчёт победителя:
- по базовому сравнению карт на руках;
- tie-break: сумма сброса;
- при повторной ничьей: всем игрокам `+1 ПО`.

- Завершение матча:
- по таблице целевых ПО (2/3/4/5-6 игроков).

### Исключить явно

- Любое поведение указов.
- Любое поведение событий.
- Любые эффекты `Знамения`.
- Любые ветки с дополнительными активными событиями.

## 3. Список карт MVP-1

Включить только базовые карты персонажей:
- `Крестьянин`
- `Стражник`
- `Разведчик`
- `Палач`
- `Дворянка`
- `Мятежник`
- `Феодал`
- `Епископ`
- `Королева`
- `Король`

Исключить:
- `Знамение`
- `Тень`

## 4. Технический скоуп backend

## 4.1 Domain

Сделать базовые модели:
- `MatchState`
- `RoundState`
- `PlayerRoundState`
- `CardInstance`

Сделать базовые rule-сервисы:
- `RoundSetupService`
- `TurnService`
- `CardEffectService` (только базовые эффекты карт)
- `RoundEndService`
- `MatchEndService`

## 4.2 Application use cases

Реализовать:
- `CreateMatch`
- `StartRound`
- `GetMatchState`
- `PlayCard`

## 4.3 Persistence

Хранить:
- состояние матча;
- состояние текущего раунда;
- зоны карт (колода, рука, сброс, отложенная);
- очки игроков.

## 4.4 API (минимум)

- `POST /matches`
- `POST /matches/{matchId}/start-round`
- `GET /matches/{matchId}`
- `POST /matches/{matchId}/play-card`

## 4.5 Realtime (минимум)

- `match_state_updated`
- `turn_changed`
- `card_played`
- `player_eliminated`
- `round_finished`
- `match_finished`

## 5. План работ по шагам

### Шаг 1. Каркас матча и раунда

- Создать domain-модели и начальные use case.
- Сделать создание матча и старт первого раунда.
- Отдавать snapshot состояния через `GET /matches/{id}`.

Критерий готовности:
- матч создаётся;
- раунд стартует;
- состояние читается API.

### Шаг 2. Базовый turn loop

- Реализовать порядок хода:
- добор;
- розыгрыш;
- передача хода.

Критерий готовности:
- ходы идут по кругу живых игроков без пропусков/циклических ошибок.

### Шаг 3. Базовые эффекты персонажей

- Реализовать эффекты карт из MVP-набора.
- Добавить валидации целей и недопустимых действий.

Критерий готовности:
- каждая карта из списка применяет базовый эффект детерминированно.

### Шаг 4. Завершение раунда и ПО

- Завершение по одному живому.
- Завершение по исчерпанию колоды.
- Подсчёт победителя и начисление ПО.

Критерий готовности:
- раунд завершается корректно в обоих сценариях.

### Шаг 5. Завершение матча

- Проверка достижения целевых ПО.
- Фиксация победителя матча.

Критерий готовности:
- матч корректно переходит в `finished`.

### Шаг 6. Realtime минимальный слой

- Публикация событий состояния в WS.
- Синхронизация клиентов при ходе/раунде/финише.

Критерий готовности:
- подключённый клиент видит изменения в реальном времени.

## 6. Тестовый контур MVP-1

## 6.1 Unit

- setup раунда;
- порядок ходов;
- базовые эффекты каждой карты;
- завершение раунда;
- начисление ПО и завершение матча.

## 6.2 Integration

- сценарий полного раунда 2 игрока;
- сценарий полного раунда 4 игрока;
- завершение по пустой колоде;
- матч до победы по целевым ПО.

## 6.3 Contract

- ответы API на `create/start/get/play`;
- WS payload на ключевые события.

## 7. Риски этапа

- Сложность эффектов отдельных карт даже без указов/событий.
- Ошибки в turn-order при множественных выбываниях.
- Неверная синхронизация состояния между API и WS.

## 8. Критерий завершения MVP-1

Этап считается завершённым, когда:
- можно сыграть матч от старта до победителя;
- используются только базовые карты персонажей (без `Знамения`, без `Тени`);
- указы и события полностью выключены и не участвуют в логике;
- API и WS дают консистентное состояние;
- тесты MVP-1 зелёные.
