# Техническое задание: Игровой движок Letter: No Mercy

## 1. Статус и цель

- Статус: актуальное ТЗ для реализации backend+realtime игрового слоя.
- Основа правил: [game-rules-summary-ru.md](/home/sshmik@wooppay.team/sites/stories-api/docs/game-rules-summary-ru.md:1) и [game-rules-qa-ru.md](/home/sshmik@wooppay.team/sites/stories-api/docs/game-rules-qa-ru.md:1).
- Цель: реализовать детерминированный движок партии/раунда без изменения уже зафиксированной игровой логики.

## 2. Область реализации

### Входит

- Модель матча и раунда.
- Ротация указов.
- Обработка карт персонажей, указов и событий.
- Токены/жетоны и модификаторы.
- Подсчёт ПО, завершение раунда/матча.
- HTTP API для игровых действий.
- WebSocket события синхронизации.

### Не входит

- Балансировка правил.
- Хоумрул-параметризация.
- Изменение существующей логики лобби/комнат.

## 3. Термины и сущности

### Матч

Партия до достижения целевых ПО:
- 2 игрока: 7 ПО
- 3 игрока: 6 ПО
- 4 игрока: 5 ПО
- 5-6 игроков: 4 ПО

### Раунд

Независимый цикл внутри матча:
- подготовка;
- ходы игроков;
- завершение и начисление ПО.

### Игрок в матче

- `alive`: в игре текущего раунда или выбыл.
- `hand`: карта в руке.
- `discard`: сброс раунда.
- `points`: победные очки матча.
- `tokens`: активные жетоны.

### Карты

- `character` (персонажи, включая `Знамение`).
- `edict` (указы, включая общие).
- `event` (события).

### Жетоны

- `FEUDAL_MARK`
- `BLACK_ROSE`
- `QUEEN_MERCY`

## 4. Архитектура игрового слоя

Рекомендуемая структура:


## 6. Инварианты движка

- В обычном режиме в раунде активно 1 событие; максимум 2 с учётом `Братоубийства`.
- Для персонажа действует только один указ одновременно.
- Лимит активных указов: 3.
- При розыгрыше персонажа указ заменяет базовый эффект карты персонажа.
- Эффекты событий имеют высший приоритет.
- Применение эффекта карты атомарно.
- При ничьей в конце раунда начисляется `+1 ПО` всем игрокам.

## 7. State-machine раунда

### 7.1 Setup

1. Удалить `Тень` из стартовой колоды (по правилам матча).
2. Перемешать колоды.
3. Отложить случайную карту персонажа.
4. Раздать стартовые карты.
5. Заменить стартовое `Знамение`, если выпало.
6. Выполнить шаг ротации указов лидером.
7. Перевести раунд в `active`.

### 7.2 Turn loop (active)

1. Текущий игрок разыгрывает карту.
2. Движок вычисляет итоговый эффект:
- активные события;
- замещение указом (если применимо).
3. Применяет эффект атомарно.
4. Проверяет условия завершения раунда:
- остался один живой;
- взята последняя карта колоды и текущий ход завершён;
- событие завершает раунд немедленно.
5. Если раунд не завершён — передаёт ход следующему живому игроку.

### 7.3 Finish

- Рассчитать победителя/победителей по правилам.
- Начислить ПО.
- Обновить лидера.
- Проверить завершение матча.

## 8. Правила завершения и tie-break

- Если живым остался 1 игрок: стандартная победа раунда.
- Если колода персонажей исчерпана: ход доигрывается, затем сравнение.
- `Эхо прошлого`: немедленное завершение, участвуют все (включая выбывших).
- Tie-break:
- сравнение старшей карты;
- при равенстве — сумма сброса;
- при повторном равенстве — всем игрокам `+1 ПО`.

## 9. События и особые кейсы

### 9.1 Знамение

- Активируется немедленно.
- Не считается обычным “разыгрышем карты за ход”.
- Карта уходит в отдельную отложенную зону до следующего раунда.
- Игрок добирает карту и продолжает ход.

### 9.2 Братоубийство

- Может активировать дополнительное событие.
- Допсобытие применяется в текущем раунде.

### 9.3 Буря

- Собрать карты рук оставшихся + отложенную карту.
- Перемешать, раздать по одной.
- Лишнюю убрать из игры.
- Сбросить временные эффекты и жетоны на картах.

### 9.4 Возврат карт в игру

Поддержать оба алгоритма (по тексту конкретного события):

1. Групповой:
- каждый игрок выбирает 1 карту из своего сброса;
- плюс отложенная карта;
- всё замешивается в колоду.

2. При выбывании:
- выбывший выбирает 1 карту из своего сброса;
- добрать до 2 верхних карт колоды (если есть);
- замешать и вернуть наверх колоды.

## 10. HTTP API

Префикс: `/matches`

- `POST /matches` — создать матч для комнаты.
- `POST /matches/{matchId}/start-round` — старт раунда.
- `GET /matches/{matchId}` — состояние матча (redacted per viewer).
- `POST /matches/{matchId}/play-card` — розыгрыш карты.
- `POST /matches/{matchId}/decision` — ответы на обязательные выборы.

Принцип:
- все действия возвращают `state snapshot + pending decisions`.

## 11. WebSocket события

Минимальный набор:

- `match_state_updated`
- `round_started`
- `turn_changed`
- `card_played`
- `player_eliminated`
- `event_activated`
- `edict_changed`
- `points_awarded`
- `round_finished`
- `match_finished`

Требование:
- snapshot персонализирован по подписчику (скрытая информация не раскрывается).

## 12. Формат pending decisions

Типы минимум:

- `choose_target`
- `choose_card_name`
- `choose_from_discard`
- `choose_edict_replace`
- `confirm_optional_effect`

Контракт:
- у decision всегда есть `actorPlayerId`, `type`, `options`, `expiresAt?`.

## 13. Persistence

Нужно хранить:

- матч, раунд, игроки матча;
- состояние колод и зон карт;
- активные указы/события;
- жетоны и модификаторы;
- журнал действий (для отладки и восстановления).

Рекомендация:
- snapshot + action log.

## 15. Тестирование

Обязательные наборы:

1. Unit-тесты правил:
- указ заменяет эффект карты;
- события приоритетнее;
- Знамение flow;
- tie-break правила;
- лимит двух событий.

2. Integration-тесты use case:
- полная симуляция раунда;
- завершение по одному выжившему;
- завершение по пустой колоде;
- сценарии возврата карт.

3. Contract-тесты API/WS:
- структура snapshot;
- pending decisions;
- redaction скрытой информации.

- Движок детерминированно отрабатывает раунд по согласованным правилам.
- Матч корректно завершается по целевым ПО.
- Реалтайм-состояние консистентно для всех участников.
- Скрытая информация не утечёт через API/WS.
- Unit + integration + contract тесты покрывают ключевые сценарии.
