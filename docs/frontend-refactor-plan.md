# Frontend Architecture Notes

## Цель рефакторинга

Привести frontend к состоянию, в котором код:

- читается по пользовательским сценариям, а не по случайным гигантским файлам;
- имеет явные границы между UI, состоянием, сетевым слоем и realtime;
- остаётся простым и понятным без смены стека и без лишнего framework-overengineering;
- подходит для дальнейшего роста проекта под более сложные room/game сценарии.

Ключевой инвариант рефакторинга:

- поведение приложения не меняется ради архитектуры;
- меняется структура кода и его поддерживаемость.

## Текущее состояние

После рефакторинга frontend организован по pragmatic feature-oriented схеме с лёгким разделением ответственности.

Актуальная структура:

```text
frontend/src/
  main.js
  app/
    api.js
    bootstrap/
      session.js
      start-app.js
    events.js
    features/
      auth/
        auth-events.js
      common/
        common-events.js
        ui-state.js
      lobbies/
        lobby-events.js
      profile/
        profile-events.js
      realtime/
        realtime.js
      rooms/
        room-flow.js
    services/
      feedback.js
      socket.js
      storage.js
    store/
      mutations.js
      selectors.js
    ui/
      layout.js
      modals.js
      render-app.js
      screens/
        home-screen.js
        lobbies-screen.js
        profile-screen.js
        room-screen.js
      shared/
        view-helpers.js
    state.js
    views.js
```

## Что было сделано

### 1. Разрезан монолитный events-слой

Раньше `frontend/src/app/events.js` был большим смешанным файлом, в котором одновременно жили:

- DOM bindings;
- auth сценарии;
- lobby сценарии;
- room сценарии;
- chat;
- websocket/realtime;
- UI state transitions;
- toast/status handling.

Теперь:

- `events.js` является только совместимым barrel-файлом;
- feature-сценарии разнесены по отдельным модулям;
- realtime вынесен из click handlers;
- feedback/service код больше не смешан с feature orchestration.

### 2. Разрезан монолитный views-слой

Раньше `views.js` рендерил всё приложение из одного файла.

Теперь:

- `layout.js` отвечает за общую композицию страницы;
- `modals.js` содержит модальные окна;
- `screens/*` содержат рендер конкретных экранов;
- `view-helpers.js` содержит общие view helper'ы.

Это сильно упростило навигацию по UI-коду.

### 3. Упрощён bootstrap

Раньше `main.js` сам знал:

- как грузить пользователя;
- как восстанавливать комнату;
- как грузить лобби;
- как подключать realtime;
- как рендерить и биндить приложение.

Теперь:

- `main.js` только запускает приложение;
- `bootstrap/session.js` отвечает за initial data/session restore;
- `bootstrap/start-app.js` собирает запуск;
- `ui/render-app.js` отвечает за render + bind цикл.

### 4. Нормализована работа со state

Раньше изменение состояния было хаотичным и шло прямыми присваиваниями в разных частях проекта.

Теперь:

- введены `store/mutations.js`;
- введены `store/selectors.js`;
- ключевые операции со state получили явные имена;
- побочные эффекты вокруг `localStorage` централизованы;
- прямые записи в state заметно сокращены.

### 5. Вынесены низкоуровневые сервисы

Теперь отдельно живут:

- `feedback.js` — status/toast/field errors;
- `socket.js` — низкоуровневая отправка websocket-сообщений;
- `storage.js` — работа с `localStorage`.

Это уменьшило связность feature-кода с браузерными деталями.

## Архитектурные правила

### 1. UI не должна знать про API и websocket

UI-слой должен:

- читать state;
- возвращать HTML;
- использовать только view helper'ы;
- не ходить в API;
- не принимать orchestration-решения.

### 2. Feature-модули должны быть локально читаемыми

Если нужно понять room-flow, разработчик должен читать:

- `features/rooms/room-flow.js`
- `features/realtime/realtime.js`
- `ui/screens/room-screen.js`
- `ui/modals.js`

а не искать логику по всему проекту.

### 3. Все низкоуровневые внешние зависимости должны проходить через services

Сюда относятся:

- websocket transport;
- localStorage;
- toast/status feedback.

Feature-модуль не должен напрямую плодить такие детали, если уже есть общий service.

### 4. Изменение состояния должно быть именованным

Плохо:

- `state.foo = ...` в случайной точке кода.

Нормально:

- `setActiveRoom(...)`
- `clearActiveRoom()`
- `setRoomModalState(...)`
- `setLobbyFilters(...)`
- `setToken(...)`

### 5. Derived logic должна уходить в selectors

Плохо:

- заново вычислять в разных местах, кто owner, кто spectator и является ли комната текущей.

Нормально:

- `isRoomOwner()`
- `getMyParticipant()`
- `getPlayers()`
- `isSameActiveRoom(roomId)`

### 6. Barrel-файлы допустимы только как совместимый вход

`events.js` и `views.js` допустимы как:

- точка совместимого импорта;
- короткий реэкспорт.

Они не должны снова превращаться в места, где живёт реальная логика.

## Что здесь применено

### Pragmatic Feature-Oriented Architecture

Почему применено:

- проект уже не является “просто страницей”;
- есть auth, lobby, room management, chat, realtime и сложные пользовательские флоу;
- при этом проект ещё не требует тяжёлого framework-слоя.

Плюсы:

- код читается по сценариям;
- локальнее изменения;
- легче добавлять новые feature-модули;
- меньше шансов снова собрать гигантский файл “на всё”.

Минусы:

- не всегда идеально ясно, где заканчивается feature и начинается shared/service;
- при плохой дисциплине можно начать плодить слишком много мелких файлов.

### Thin Bootstrap

Почему применено:

- точка входа не должна знать бизнес-детали приложения.

Плюсы:

- запуск приложения проще понимать;
- легче тестировать и менять инициализацию;
- `main.js` не превращается в второй монолит.

Минусы:

- появляется дополнительный слой файлов `bootstrap/*`;
- нужно удерживать дисциплину, чтобы туда не уехала feature-логика.

### Explicit Mutations / Selectors

Почему применено:

- frontend уже дорос до состояния, где хаотичный state начинает ломать читаемость.

Плюсы:

- изменение состояния заметнее и предсказуемее;
- меньше дублирования логики;
- проще рефакторить сценарии без скрытых side effects.

Минусы:

- код становится чуть более многословным;
- есть риск переусложнить слой state, если делать mutation на каждую мелочь без здравого смысла.

### Service Extraction

Почему применено:

- browser/storage/socket/toast детали не должны течь по всем feature-модулям.

Плюсы:

- меньше связность;
- понятнее место для общих низкоуровневых зависимостей;
- удобнее менять реализацию без затрагивания feature-кода.

Минусы:

- всегда есть риск превратить `services` в новую свалку;
- нужно жёстко контролировать, что туда попадает только действительно низкоуровневая логика.

## Что считается хорошей практикой дальше

### Для новых feature-сценариев

Новый пользовательский сценарий должен раскладываться так:

- orchestration в `features/...`;
- UI в `ui/...`;
- внешний transport в `services/...`;
- изменение состояния через `store/mutations.js`;
- derived-чтение через `store/selectors.js`, если логика повторяется.

### Для новых view-функций

View должна:

- быть чистой по отношению к данным;
- не делать `fetch`;
- не ходить в websocket;
- не писать в state;
- не знать, где именно хранятся токен, сокет и т.п.

### Для новых state-изменений

Если изменение:

- повторяется;
- затрагивает несколько полей;
- связано с `localStorage`;
- влияет на согласованность UI;

его лучше оформлять как mutation.

### Для selectors

Selector стоит вводить, если:

- логика повторяется в нескольких модулях;
- это derived-state, а не raw field access;
- его имя делает код заметно понятнее.

Не нужно делать selector на каждый примитивный доступ к одному полю.

## Что не нужно делать дальше

- не возвращать бизнес-логику в `events.js` и `views.js`;
- не складывать feature-код в `services`;
- не делать generic helper'ы “на всё”;
- не плодить абстракции без конкретной пользы;
- не превращать `store/mutations.js` в новую мусорную кучу без структуры.

## Что ещё сознательно оставлено простым

Рефакторинг финализирован, но без фанатизма.

Сознательно не вводились:

- полноценный state manager;
- class-based architecture;
- отдельный event bus;
- routing framework;
- сложный dependency injection на frontend.

Для текущего проекта это было бы избыточно.

## Как оценивать код дальше

Если новый фронтендер откроет проект, хороший результат должен ощущаться так:

- понятно, где искать room/lobby/auth/realtime логику;
- UI, state и transport не смешаны в одном месте;
- entrypoint короткий и предсказуемый;
- новые сценарии можно добавлять локально;
- код выглядит дисциплинированным, но не переусложнённым.

Именно это состояние и считается целевым для текущего frontend-слоя.
