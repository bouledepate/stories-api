# Руководство По Поддерживаемому Коду

## Цель

Этот документ нужен как практическое правило для дальнейшей разработки в проекте.

Главная идея:
- код должен быть понятным по структуре;
- изменение одного сценария не должно расползаться по всему проекту;
- архитектура должна помогать читать и менять код, а не мешать.

## Базовый принцип

Если новый код не делает систему понятнее, локальнее и предсказуемее, значит решение выбрано плохо, даже если оно “технически работает”.

## Текущая архитектура

Проект использует слоистую структуру:

```text
src/
  Presentation/
  Application/
  Domain/
  Infrastructure/
  Shared/
```

## Ответственность слоёв

### `Presentation`

Здесь живёт всё, что связано с входом в систему извне:
- HTTP actions;
- middleware;
- responders;
- web entrypoints.

Что здесь можно делать:
- читать request;
- валидировать вход;
- вызывать application handler;
- превращать результат в HTTP response.

Что здесь нельзя делать:
- писать бизнес-логику;
- выполнять SQL;
- принимать архитектурные решения за domain/application;
- держать сложную orchestration логику.

Правило:
- action должен быть тонким;
- если action растёт, значит логика ушла не туда.

### `Application`

Это слой сценариев.

Здесь живёт:
- orchestration use case;
- request/result модели конкретного use case;
- координация domain ports;
- прикладные проверки сценария.

Правило структуры:
- каждый use case живёт в своей папке;
- рядом лежат `Handler`, `Request`, `Result`, если они принадлежат только ему.

Пример:

```text
Application/
  Rooms/
    CreateRoom/
      CreateRoomHandler.php
      CreateRoomRequest.php
```

Что здесь можно делать:
- вызывать repository interfaces;
- собирать result object;
- описывать сценарий “что происходит по шагам”.

Что здесь нельзя делать:
- лезть в HTTP API Slim напрямую;
- читать заголовки или route params;
- писать DBAL-запросы;
- смешивать всё в один огромный service.

### `Domain`

Это слой бизнес-смысла и контрактов.

Здесь живёт:
- модели;
- repository interfaces;
- domain-level контракты.

Что здесь нельзя:
- зависеть от HTTP;
- зависеть от framework;
- знать про DBAL, Ratchet, Slim, env.

### `Infrastructure`

Это реализация технических адаптеров.

Здесь живёт:
- persistence;
- websocket server;
- connection factories;
- config adapters.

Правило:
- infrastructure реализует контракты;
- infrastructure не должен управлять бизнес-сценарием.

### `Shared`

Это не “всё общее подряд”.

Здесь должны жить только действительно общие вещи:
- общие error types;
- security primitives;
- небольшие утилиты и нормализаторы.

Что нельзя складывать в `Shared`:
- HTTP-specific classes;
- middleware;
- websocket delivery;
- DB connection layer;
- код “пока не знаю куда положить”.

Если хочется положить что-то в `Shared`, сначала надо ответить:
- это правда переиспользуемый примитив?
- он не относится к `Presentation`, `Application` или `Infrastructure`?

Если ответ неочевиден, почти наверняка `Shared` не подходит.

## Правила написания нового кода

### 1. Начинай с use case

Если добавляется новая возможность, сначала нужно понять:
- это новый сценарий?
- в каком модуле он живёт?

После этого создаётся директория:

```text
Application/<Module>/<UseCase>/
```

А уже потом:
- `Handler`
- `Request`
- `Result`
- transport entrypoint
- infrastructure changes

Не наоборот.

### 2. Держи локальность

Если request/result нужны только одному сценарию, они не должны жить в общей папке.

Плохо:

```text
Application/Rooms/Dto/*
Application/Rooms/Result/*
Application/Rooms/Handlers/*
```

Хорошо:

```text
Application/Rooms/JoinRoom/
  JoinRoomHandler.php
  JoinRoomRequest.php
  JoinRoomResult.php
```

Это уменьшает “прыжки глазами” по проекту.

### 3. Предпочитай явные result object’ы

Если use case возвращает значимую структуру, лучше использовать явный result object, а не массив.

Массив допустим только если:
- ответ совсем маленький;
- структура действительно одноразовая;
- typed object явно не окупается.

Нужно делать result object, если:
- ответ используется в нескольких местах;
- структура уже не помещается в голову сразу;
- появляются вложенные поля;
- тесты начинают проверять кучу строковых ключей.

### 4. Не раздувай handler

Handler должен описывать сценарий.

Если в нём уже:
- много ветвлений;
- много повторяющихся проверок;
- много вспомогательной логики;

нужно выносить:
- общий support/helper внутри модуля;
- отдельный result object;
- domain contract;
- infrastructure adapter.

Но не надо выносить абстракции “на будущее” без причины.

### 5. Support-классы использовать аккуратно

`Support` допустим только если:
- логика реально используется несколькими use case;
- эта логика не belongs to one specific handler.

`Support` превращается в мусорку очень быстро.

Нельзя складировать туда:
- случайные helper methods;
- чужую бизнес-логику;
- “временные штуки”.

Если метод нужен одному handler, он должен жить либо:
- внутри handler;
- либо в более точном классе рядом с этим handler.

### 6. Тонкий transport

`Presentation` слой должен быть максимально тупым:
- прочитать request;
- собрать request object;
- вызвать handler;
- вернуть response.

Если в action появляется мысль:
- “сейчас я тут ещё немного бизнес-логики добавлю”

это почти всегда ошибка.

### 7. Не смешивай transport и validation

HTTP validation может жить рядом с transport/use case boundary, но не должна протекать в domain model.

Правильная последовательность:
1. request пришёл в `Presentation`
2. собрали request object
3. провалидировали вход
4. вызвали handler

### 8. Repository interface должен быть осмысленным

Не делай generic repository “на всё”.

Плохо:
- `save(array $data)`
- `findAllBy(array $filters)`

Хорошо:
- `findOwnedRoomId(string $ownerUserId): ?string`
- `inviteCodeExists(string $inviteCode): bool`

Интерфейс должен выражать смысл сценария, а не абстрактную CRUD-механику.

### 9. Infrastructure не должен выглядеть как application

Infrastructure отвечает на вопрос:
- как это сделать технически?

Application отвечает на вопрос:
- что должно произойти по сценарию?

Если в infrastructure начинают появляться сценарные решения, граница сломана.

### 10. Не использовать `Shared` как свалку

Если непонятно, куда положить класс, нельзя автоматически выбирать `Shared`.

Сначала проверить:
- это transport?
- это use case?
- это infrastructure?
- это domain contract?

Только если ни один из этих ответов не подходит, можно думать про `Shared`.

## Признаки хорошего кода в этом проекте

- По дереву каталогов понятно, где искать нужный сценарий.
- Чтобы изменить один use case, не нужно открывать десять случайных файлов.
- HTTP action короткий.
- Handler читается сверху вниз как сценарий.
- SQL живёт только в infrastructure.
- Ошибки выражены явно.
- Ответы сценариев либо typed, либо очень простые.
- `Shared` остаётся маленьким.

## Признаки архитектурного запаха

- Появляется новый “общий service”, который знает слишком много.
- В `Shared` начинают оседать middleware, helper-классы, feature-specific вещи.
- В `Application` снова появляются общие `Dto`, `Result`, `Utils` на весь модуль без жёсткой необходимости.
- Один handler начинает делать половину модуля.
- Transport-код начинает принимать бизнес-решения.
- Infrastructure-код начинает определять сценарий, а не только реализовывать его.
- Новый код возвращает огромный `array<string, mixed>` с неочевидной структурой.

## Когда допустим компромисс

Иногда строгая архитектура не окупается.

Можно упростить, если:
- сценарий маленький;
- структура ответа тривиальна;
- изменение локальное и не тянет новых зависимостей;
- дополнительный слой только увеличит шум.

Но компромисс должен быть осознанным и локальным.

Нельзя:
- превращать временное решение в новую норму;
- копировать плохой компромисс дальше по проекту.

## Рекомендации для следующих изменений

- Для новых модулей сразу использовать `Presentation / Application / Domain / Infrastructure`.
- Для каждого нового use case сначала создавать директорию сценария.
- Если сценарий возвращает snapshot или список с вложенностью, сразу думать о result object.
- Перед добавлением нового класса в `Shared` сначала проверить, нельзя ли положить его ближе к реальному слою.
- Перед добавлением нового helper-класса спросить себя, не скрывает ли он плохую структуру handler’а.

## Короткое правило

Хороший код в этом проекте должен отвечать на три вопроса без раскопок:
- где находится сценарий;
- где находится transport;
- где находится техническая реализация.

Если для ответа приходится “знать проект наизусть”, архитектура начала деградировать.
